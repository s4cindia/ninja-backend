Implement US-PDF-4.1: PDF Audit API Endpoints

Create REST API endpoints for PDF accessibility auditing.

File to create: src/routes/pdf.routes.ts

Requirements:

1. Upload and audit endpoint:
   POST /pdf/audit-upload
   - Accept multipart/form-data with PDF file
   - Validate file is PDF (check MIME type and magic bytes)
   - Max file size: 100MB
   - Create job record in database
   - Queue audit job
   - Return: { jobId, status: 'queued' }

2. Get audit status:
   GET /pdf/job/:jobId/status
   - Return current job status
   - Include progress percentage if available
   - Return: { jobId, status, progress, createdAt, updatedAt }

3. Get audit results:
   GET /pdf/job/:jobId/audit/result
   - Return full audit results when complete
   - Return 202 if still processing
   - Return: PdfAuditResult (from pdf.types)

4. Get ACR report:
   GET /pdf/job/:jobId/acr
   - Generate and return ACR report
   - Query params: format=json|html
   - Return: ACRReport

5. Download report:
   GET /pdf/job/:jobId/report
   - Download audit report as PDF or DOCX
   - Query params: format=pdf|docx
   - Return: File download

6. List user's audits:
   GET /pdf/audits
   - Paginated list of user's PDF audits
   - Query params: page, limit, status
   - Return: { data: PdfAuditResult[], pagination }

7. Delete audit:
   DELETE /pdf/job/:jobId
   - Soft delete audit record
   - Return: { success: true }

8. Middleware:
   - Authentication required (use existing auth middleware)
   - Rate limiting (10 uploads per minute)
   - File validation middleware

9. Error handling:
   - 400: Invalid file type, file too large
   - 401: Unauthorized
   - 404: Job not found
   - 429: Rate limit exceeded
   - 500: Server error

Reference:
- Existing EPUB routes in src/routes/epub.routes.ts
- PdfAuditService for audit execution
- Existing auth and validation middleware

Tests: Create pdf.routes.test.ts with API integration tests
